# 人工升级规则（Escalation Policy）

当证据不足以自动拦截，或场景强依赖组织上下文时，应触发人工升级流程，以降低误判与漏判成本。

## 1) 自动升级触发条件（建议）

- verdict 为 `suspicious`
- profile 为 `FAST` 且存在风险信号（证据收集有限）
- 出现以下任一高风险意图/动作：
  - 凭据验证/登录诱导（`semantic.intent == "credential_theft"`）
  - OAuth/权限变更/委派访问（协作意图集合）
  - 付款/转账/更改收款信息（`invoice_payment` 等）
- 认证失败组合：
  - DMARC fail + Reply-To mismatch + 登录 URL（对应硬规则）
- 组织策略触发：
  - 新联系人/首次发件域名（需由接入层提供）

## 2) 升级后的人工检查清单

- 核验发件人身份（内部通讯录/历史线程/IM/电话）
- 检查 URL 落地页（隔离环境打开，观察登录表单与品牌）
- 若有附件：在沙箱中静态/动态分析并提取 IOC
- 若为 BEC：核验付款/收款信息的变更是否走过审批流程

## 3) 处置分流建议

- `suspicious`：默认“先隔离再复核”或“强提示后灰度放行”，取决于业务容忍度
- `phishing`：隔离 + 事件响应（通知、溯源、封禁 IOC）
